# 设计思路

## 核心问题

AI 失忆，每次开新对话上下文全丢。

## 设计目标

1. 跨会话记忆：记住之前的对话、决策、踩坑
2. 快速搜索：快速找到之前会话中的具体内容
3. 零依赖：不依赖任何插件、服务、数据库
4. 完全可控：我写什么，它就记什么
5. 可迁移：换个 AI 工具？把文件夹复制过去就行

## 架构设计

### 三层架构

```
┌─────────────────────────────────────┐
│  知识层（决策、踩坑、偏好）          │
│  - 结构化、可复用                    │
└─────────────────────────────────────┘
           ▲
           │ 提炼
           │
┌─────────────────────────────────────┐
│  索引层（会话索引）                  │
│  - 结构化、可搜索                    │
└─────────────────────────────────────┘
           ▲
           │ 原始数据
           │
┌─────────────────────────────────────┐
│  原始层（CC Switch 会话记录）        │
│  - 完整会话、不可搜索                │
└─────────────────────────────────────┘
```

### 混合方案（实时生成 + 离线生成）

**实时生成：**
- 时机：每次会话结束时
- 方式：自动提取关键信息，生成索引
- 优点：自动化，不需要手动触发
- 缺点：每次会话都要执行

**离线生成：**
- 时机：每周执行一次
- 方式：扫描 CC Switch 的 JSONL 文件，生成索引
- 优点：可以处理所有历史会话
- 缺点：需要手动执行

### 增量更新

- 记录上次更新时间
- 只处理上次更新后新增或修改的文件
- 节省时间和 token

## 与市面方案对比

| 方案 | 优点 | 缺点 | 我们的差距 |
|------|------|------|------------|
| MCP 插件 | 人类可读、本地存储 | 多一个依赖 | 小 |
| AI 自动生成 | 最省事、零配置 | 不可控、黑盒 | 大 |
| 向量数据库 | 专业、强大 | 太重、维护成本高 | 大 |
| 云端同步 | 方便、协作 | 隐私风险、依赖网络 | 大 |
| 纯 Markdown | 零依赖、完全可控 | 缺少快速搜索 | 小 |
