# V1：知识库搭建方法论

> 一套「先画格子再放书」的知识整理流程。适用于任何领域的结构化知识管理。

## 核心思路

大多数人的知识管理方式：看到好文章 → 收藏 → 再也不看。

问题不是「收藏得不够多」，而是「没有结构」。100 篇文章散落在文件夹里，等于没有。

**我们的方法**：先建骨架，再填内容。

```
定 Schema → 建骨架 → 批量 Ingest → Lint 检查
```

## 四步流程

### Step 1：定 Schema

定义你的知识领域需要哪些维度。Schema 不是固定模板，而是你的「分类思维」。

```markdown
# Schema 示例（前端面试知识）

## 一级分类
- 核心知识体系（渲染流程、DOM、异步）
- 技术专题（JS、CSS、框架、工程化）
- 面试题库（按技术点分类）

## 二级分类
- 每个一级分类下，按主题细分
- 每个主题有：概念笔记 + 题库 + 实战场景

## 元数据
- 难度：⭐ / ⭐⭐ / ⭐⭐⭐
- 来源：原创 / 整理 / 转载
- 状态：草稿 / 完善 / 定稿
```

### Step 2：建骨架

根据 Schema 创建目录结构和空文件。

```bash
# 示例：创建知识库骨架
mkdir -p knowledge-base/{核心知识,技术专题,面试题库}
touch knowledge-base/核心知识/渲染流程.md
touch knowledge-base/技术专题/JavaScript/闭包.md
# ... 根据你的 Schema 展开
```

### Step 3：批量 Ingest

将原始文档批量导入骨架。可以是：
- 爬虫抓取（飞书、Notion、语雀等）
- 手动复制粘贴
- 脚本批量转换格式

**关键**：不是「一篇一篇整理」，而是「批量倒入，再逐个校准」。

### Step 4：Lint 检查

像代码有 ESLint 一样，知识库也需要检查：

- **完整性**：Schema 定义的维度是否都有内容？
- **一致性**：文件命名、格式、元数据是否统一？
- **交叉引用**：相关主题之间是否有链接？
- **质量**：内容是否准确、是否有过时信息？

## Schema 模板

```markdown
# [你的知识领域] Schema

## 分类体系
1. [一级分类 A]
   - [二级分类 A1]
   - [二级分类 A2]
2. [一级分类 B]
   - [二级分类 B1]

## 元数据规范
- 难度：
- 来源：
- 状态：
- 更新日期：

## 文件命名规范
- 格式：`[编号].[主题名称].md`
- 示例：`01.闭包.md`、`02.事件循环.md`

## 内容模板
- 每个文件包含：概念 → 原理 → 示例 → 面试考点
```

## 示例

查看 `examples/` 目录，有两个示例文件展示格式规范：
- `概念笔记示例.md` — 知识点的标准格式
- `题库示例.md` — 面试题的标准格式

## 适用场景

| 领域 | Schema 示例 |
|------|------------|
| 前端面试 | JS / CSS / React / Vue / 网络 / 性能优化 |
| 后端开发 | 语言 / 框架 / 数据库 / 系统设计 / DevOps |
| 产品经理 | 需求分析 / 用户研究 / 竞品分析 / 数据分析 |
| 考研复习 | 政治 / 英语 / 数学 / 专业课 |
| 法考 | 民法 / 刑法 / 行政法 / 诉讼法 |

## 为什么这样做有效

1. **降低启动成本**：先有骨架，填内容比从零开始容易
2. **保证覆盖率**：Schema 定义了「应该知道什么」，避免遗漏
3. **支持增量更新**：新知识有明确的归属位置
4. **可搜索**：结构化的知识比散落的笔记容易检索
